Exercise 49 — Streamify: Love/Hate Songs

Z Notation Modelling — Part 2 of a 5-exercise series

The Scenario

This exercise extends the Streamify base system defined in Exercise 48. Users can now express opinions about songs by loving or hating them. Each user maps to a finite set of songs they love and a finite set of songs they hate.

The key modelling challenge is ensuring that love and hate are mutually exclusive for the same user and song.

Context from Exercise 48

The base system already tracks:

  • $songs : \mathbb{F}\; SongId$ — the catalogue of songs on the platform.
  • $users : \mathbb{F}\; UserId$ — the registered users.
  • $playlists$, $playlist\_owner$, $playlist\_subscribers$ — playlist data with associated integrity constraints.

We now extend this system with two new state variables.

New Declarations

The Task

Task: Complete the schema by adding predicates that capture the following five constraints:
  1. Only users in $users$ can appear in $hated$.
  2. Only songs in $songs$ can appear in $hated$.
  3. Only users in $users$ can appear in $loved$.
  4. Only songs in $songs$ can appear in $loved$.
  5. No user can both hate and love the same song.

Part (a) — Only users in users can appear in hated

Task: Write a predicate ensuring that every user who has hated at least one song must be a registered user in $users$.

Vocab for Part (a)

TermDefinition
Referential integrityEnsuring that references to entities (songs, users, playlists) point to things that actually exist.
Domain constraintRestricting which keys can appear in a function.

Symbols for Part (a)

SymbolNamePlain English
domdomainthe set of inputs on which a function is defined
⊆subsetevery element of the left set is also in the right set

Step 1 — Understand the task

We need a domain constraint that prevents $hated$ from containing a user who does not exist. This is referential integrity on the key side of the function: no orphaned hate-lists for phantom users.

Step 2 — Identify the approach

The function $hated$ is keyed by $UserId$. The set $users$ contains all valid user IDs. We need every key of $hated$ to also be in $users$. This is a direct domain constraint: dom of the function ⊆ the reference set.

Step 3 — Plan the components

  • Extract the keys of $hated$ using dom.
  • Assert they are a subset of $users$.

Part (b) — Only songs in songs can appear in hated

Task: Write a predicate ensuring that for every user who has hated songs, each song in their hate-list must exist in the catalogue $songs$.

Vocab for Part (b)

TermDefinition
Referential integrityEnsuring that references to entities (songs, users, playlists) point to things that actually exist.
Domain constraintRestricting which keys can appear in a function.

Symbols for Part (b)

SymbolNamePlain English
∀universal quantifier"for all"
domdomainthe set of inputs on which a function is defined
•spotseparates the bound variable from the body in a quantified expression
⊆subsetevery element of the left set is also in the right set

Step 1 — Understand the task

We need referential integrity for the song values inside $hated$. Every song that any user hates must actually exist in the platform's catalogue. This prevents hate-lists from referencing phantom or deleted tracks.

Step 2 — Identify the approach

Each value of $hated$ is a set of songs ($\mathbb{F}\; SongId$), not a single song. So $\text{ran}\; hated$ would give a set of sets of songs, not individual songs — we cannot write ran hated ⊆ songs.

Instead, we must use ∀ to iterate over each user in the domain, unwrap the set-valued function, and check each hate-set individually.

Step 3 — Plan the components

  • Use ∀ to iterate over every user ID $i$ in dom hated.
  • For each user $i$, apply hated to get their hate-set: $hated\; i$.
  • Assert that set is a subset of $songs$.

Part (e) — No user can both love and hate the same song

Task: Write a predicate ensuring that for every user who has expressed any opinion (love or hate), their love-set and hate-set must be disjoint — no song can appear in both.

Vocab for Part (e)

TermDefinition
Mutual exclusivityA song cannot be in both sets for the same user.
Disjoint setsTwo sets with no elements in common (intersection is empty).
Union of domainsCombining two sets of keys to cover all relevant users.

Symbols for Part (e)

SymbolNamePlain English
∀universal quantifier"for all"
domdomainthe set of inputs on which a function is defined
∪unionthe set of elements that belong to at least one of the two sets
•spotseparates the bound variable from the body in a quantified expression
∩intersectionthe set of elements that belong to both sets simultaneously
∅empty setthe set with no elements

Step 1 — Understand the task

We need a mutual exclusivity constraint. For each user, the songs they love and the songs they hate must be disjoint: no song can simultaneously be loved and hated by the same person.

Step 2 — Identify the approach

We need to check every user who has expressed any opinion. A user might appear only in $hated$, only in $loved$, or in both. To cover all cases, we quantify over the union of domains: dom hated ∪ dom loved. For each such user, we require that the intersection of their hate-set and love-set equals the empty set.

Step 3 — Plan the components

  • Use ∀ to iterate over every user $i$ in dom hated ∪ dom loved.
  • For each user $i$, compute the intersection of their hate-set and love-set: hated i ∩ loved i.
  • Assert that intersection equals ∅.

Solution

Click to reveal

Build Step 1 — Begin with ∀ over the combined domain

SymbolPlain EnglishWhat it does here
∀universal quantifier — "for all"the constraint must hold for every user with any opinion
$i$bound variable — represents each user ID in turnstands for each user ID as we iterate
domdomain — the set of inputs on which a function is definedextracts user IDs from each function
∪union — the set of elements that belong to at least one of the two setscombines both domains to cover all users with opinions

Start with ∀ to say "for every user i", drawing from the union of both domains:

$$\forall\, i : \text{dom}\; hated \cup \text{dom}\; loved\ \ldots$$

Build Step 2 — Add the spot and access both sets for user $i$

SymbolPlain EnglishWhat it does here
•spot — separates the bound variable from the bodyconnects "for all $i$" to the condition
$hated\; i$function application — the hate-set for user $i$the set of songs user $i$ hates
$loved\; i$function application — the love-set for user $i$the set of songs user $i$ loves

After •, we access both $hated\; i$ and $loved\; i$ for the current user:

$$\forall\, i : \text{dom}\; hated \cup \text{dom}\; loved \bullet hated\; i \quad\ldots\quad loved\; i\ \ldots$$

Build Step 3 — Intersect the two sets

SymbolPlain EnglishWhat it does here
∩intersection — the set of elements that belong to both sets simultaneouslyfinds songs that appear in both the hate-set and the love-set

Use ∩ to compute the overlap between the two sets:

$$\forall\, i : \text{dom}\; hated \cup \text{dom}\; loved \bullet hated\; i \cap loved\; i\ \ldots$$

Build Step 4 — Require the intersection to be empty

SymbolPlain EnglishWhat it does here
$=$equality — the left side equals the right sideasserts the intersection has a specific value
∅empty set — the set with no elementsno songs in common — the sets are disjoint

Assert that the intersection equals the empty set — nothing in common:

$$\forall\, i : \text{dom}\; hated \cup \text{dom}\; loved \bullet hated\; i \cap loved\; i = \varnothing$$

Complete answer

$$\forall\, i : \text{dom}\; hated \cup \text{dom}\; loved \bullet hated\; i \cap loved\; i = \varnothing$$

"For every user $i$ who has expressed any opinion, the intersection of their hate-set and love-set must be the empty set — no song can be both loved and hated by the same user."

Why dom hated ∪ dom loved? We quantify over the union of domains to cover every user who has either a hate-list or a love-list (or both). A user who only appears in one domain trivially satisfies the constraint: the missing set is empty, so the intersection is empty.

Complete Definition — Full Schema

Assembling both declarations and all five constraints into a single schema:

StreamifyLoveHate (extends Streamify)
$hated : UserId \;{\pfun}\; \mathbb{F}\; SongId$
$loved : UserId \;{\pfun}\; \mathbb{F}\; SongId$
$\text{dom}\; hated \subseteq users$    ...(a)
$\forall\, i : \text{dom}\; hated \bullet (hated\; i) \subseteq songs$    ...(b)
$\text{dom}\; loved \subseteq users$    ...(c)
$\forall\, i : \text{dom}\; loved \bullet (loved\; i) \subseteq songs$    ...(d)
$\forall\, i : \text{dom}\; hated \cup \text{dom}\; loved \bullet hated\; i \cap loved\; i = \varnothing$    ...(e)

Reading the schema: Above the line are the two new variable declarations. Below the line are the five constraints — the invariants. Constraints (a) and (c) enforce domain constraints on users. Constraints (b) and (d) enforce referential integrity on songs. Constraint (e) enforces mutual exclusivity between love and hate.

Summary Table

(b), (d) ∀ $i$ : dom $f$ • $f\; i$ ⊆ $S$ Referential integrity — the function maps to sets, and every element of those sets must come from a reference set.
(e) ∀ $i$ : $D$ • $A\; i$ ∩ $B\; i$ = ∅ Mutual exclusivity — two set-valued functions must be disjoint for each key.
