Exercise 52 — Streamify: Playing Songs

Software Engineering Mathematics — Z Notation Modelling (Part 5 of a 5-exercise series)

The Scenario

We now consider users playing songs. To do this, we introduce timestamps, user status (premium vs. standard), and recursive functions that filter and aggregate play records.

New Types and Functions

New type definitions:

$$Tstamp == \mathbb{N} \quad\text{(a timestamp, represented as a natural number)}$$ $$Status ::= premium \mid standard$$ $$Play == SongId \times UserId \times Tstamp$$

New function:

$$user\_status : UserId \pfun Status$$

$user\_status$ assigns each user a fixed status (premium or standard). It does not change.

Reading a Play tuple: A value $x : Play$ is a triple $(s, u, t)$ where:

  • $x.1 = s$ is the $SongId$ of the song that was played.
  • $x.2 = u$ is the $UserId$ of the user who played it.
  • $x.3 = t$ is the $Tstamp$ when it was played.

Context from Exercises 48–51

The following are assumed to exist from prior exercises: songs, users, playlists, hated, loved, length ($SongId \pfun \mathbb{N}$, giving song length in seconds), popularity, user_status.

Step 1 — Understand the task

We need a recursive filter that walks through a sequence of Play records and keeps only those where the user has status premium. The output is a sequence of the same type — just with standard-user plays removed.

Step 2 — Identify the approach

Every recursive filter on sequences follows the same skeleton:

  • Base case: what happens with the empty sequence?
  • Recursive case: decompose into head and tail, check the head's user status, keep or skip, then recurse on the tail.

Step 3 — Plan the components

Base case: filtering an empty sequence yields an empty sequence.

Recursive case: the head $x$ is a Play triple. Apply user_status to x.2 (the UserId). If the result is premium, keep $x$ by prepending it. If standard, skip $x$ and recurse on the tail alone.

We need a recursive filter that is the mirror of premium_plays: it walks through a sequence of Play records and keeps only those where the user has status standard. Premium-user plays are discarded.

Step 2 — Identify the approach

The structure is identical to Part (a). The only difference is which status value triggers "keep" and which triggers "skip" — the roles of premium and standard are swapped.

  • Base case: empty sequence returns empty sequence (same as Part a).

Step 3 — Plan the components

Recursive case: the head $x$ is a Play triple. Apply user_status to x.2. If the result is standard, keep $x$. If premium, skip $x$.

Step 1 — Understand the task

We need a recursive function that walks through a sequence of Play records and sums up the duration of every song played. Unlike Parts (a) and (b), which filtered and returned sequences, this function returns a single natural number — it is an accumulator rather than a filter.

Step 2 — Identify the approach

The same recursive skeleton applies:

  • Base case: what is the total length of zero plays? Zero seconds.
  • Recursive case: decompose into head and tail. Extract the song's length from the head, add it to the recursive total of the tail.

Step 3 — Plan the components

Base case: the total of zero plays is 0 seconds.

Recursive case: the head $x$ is a Play triple. Its first component x.1 is the SongId. Apply length to x.1 to get the song's duration in seconds. Add that to the accumulator — the recursive total of the remaining plays.
