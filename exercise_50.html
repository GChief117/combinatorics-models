Exercise 50 — Streamify: Deriving Information

Software Engineering Mathematics — Z Notation Modelling (Part 3 of a 5-exercise series)

The Scenario

Using the Streamify model built in Exercise 48 (playlists, ownership, subscriptions) and Exercise 49 (love/hate), we now derive useful information from the system state. This exercise practises reading and querying a Z specification — writing expressions that extract specific facts from the model's variables.

Context — Available Definitions from Exercises 48 and 49

From the combined Streamify model, we have:

From Exercise 48 (base system):
$songs : \mathbb{F}\; SongId$ — all songs on the platform
$users : \mathbb{F}\; UserId$ — all registered users
$playlists : PlaylistId \;{\pfun}\; Playlist$ — playlist contents (where $Playlist == \text{iseq}_1\; SongId$)
$playlist\_owner : PlaylistId \;{\pfun}\; UserId$ — who owns each playlist
$playlist\_subscribers : PlaylistId \;{\pfun}\; \mathbb{F}_1\; UserId$ — who subscribes to each playlist

From Exercise 49 (love/hate):
$hated : UserId \;{\pfun}\; \mathbb{F}\; SongId$ — songs each user hates
$loved : UserId \;{\pfun}\; \mathbb{F}\; SongId$ — songs each user loves

Part (a) — Users Who Have Subscribed to No Playlists

Task: Find all registered users who do not appear as a subscriber of any playlist.

Vocab for Part (a)

TermDefinition
Set difference (\)Elements in A but not in B. $A \setminus B$ removes from A everything that also appears in B.
Generalised union ($\bigcup$)Flattens a set of sets into one set. Collects every element that appears in any of the inner sets.
Range (ran)The set of all output values of a function. When the function maps to sets, the range is a set of sets.

Symbols for Part (a)

SymbolNamePlain English
usersstate variablethe full set of all registered users
$\setminus$set difference$A \setminus B$ — elements of A not in B
$\bigcup$generalised unionflattens a set of sets into one set
ranrangethe set of all output values of a function
playlist_subscribersstate variablemaps each playlist ID to its non-empty set of subscribers

Step 1 — Understand the task

We need the set of all registered users who do not subscribe to any playlist. The subscription data lives inside playlist_subscribers, which maps each playlist to a set of user IDs.

Step 2 — Identify the approach

Strategy: first collect all users who subscribe to at least one playlist, then subtract them from the full users set using set difference $\setminus$.

The subscriber data is buried inside the values of playlist_subscribers. Each value is a set of users. So we need ran to get those sets, then $\bigcup$ to flatten them into a single set.

Step 3 — Plan the components

  1. Start with users — the full user set.
  2. Build the "all subscribers" set: ran playlist_subscribers gives a set of sets; $\bigcup$ flattens it.
  3. Subtract: $\setminus$ removes the subscribers from the full set.

Solution

Build Step 1 — Start with the full user set

SymbolPlain EnglishWhat it does here
usersstate variable — the full set of all registered usersour starting pool — everyone on the platform

We begin with users, the set of every registered user. We will subtract from this:

$$users \;\setminus\; \ldots$$

Build Step 2 — Build the set of all subscribers: $\bigcup$(ran playlist_subscribers)

SymbolPlain EnglishWhat it does here
ranrange — the set of all output values of a functioncollects all the subscriber sets (one per playlist)
$\bigcup$generalised union — flattens a set of sets into one setmerges all subscriber sets into one big set of user IDs

ran playlist_subscribers gives the collection of all subscriber sets — one set per playlist. For example:

$$\text{ran}\; playlist\_subscribers = \{\{u_1, u_2\},\; \{u_1, u_3\},\; \{u_2\}\}$$

$\bigcup$ flattens these into a single set of all users who subscribe to at least one playlist:

$$\bigcup(\text{ran}\; playlist\_subscribers) = \{u_1, u_2, u_3\}$$

Our expression so far:

$$users \;\setminus\; \bigcup(\text{ran}\; playlist\_subscribers)$$

Build Step 3 — Apply set difference $\setminus$

SymbolPlain EnglishWhat it does here
$\setminus$set difference — elements of A not in Bremoves all subscribers from the full user set

$\setminus$ removes the subscriber set from users. If $users = \{u_1, u_2, u_3, u_4, u_5\}$, then:

$$A = \{u_1, u_2, u_3, u_4, u_5\} \;\setminus\; \{u_1, u_2, u_3\} = \{u_4, u_5\}$$

These are the users who have not subscribed to any playlist.

Complete answer

$$A == users \;\setminus\; \bigcup(\text{ran}\; playlist\_subscribers)$$

"All registered users minus everyone who appears in at least one playlist's subscriber list."

Part (b) — Playlists with at Least 100 Subscribers

Task: Find all playlist IDs where the subscriber count is 100 or more.

Vocab for Part (b)

TermDefinition
Set comprehensionAn expression $\{\ \text{decl}\ |\ \text{filter}\ \}$ that builds a set by iterating over a source and keeping elements that satisfy a predicate.
Cardinality (#)The number of elements in a finite set. $\#\{a, b, c\} = 3$.
Domain (dom)The set of all input values for which a function is defined.

Symbols for Part (b)

SymbolNamePlain English
$\{\ |\ \}$set comprehensioniterate over a source, keep elements satisfying a predicate
domdomainthe set of all input values of a function
playlist_subscribersstate variablemaps each playlist ID to its non-empty set of subscribers
#cardinalitythe number of elements in a set
$\geq$greater than or equalnumeric comparison: left side is at least as large as right side

Step 1 — Understand the task

We need a set comprehension that collects playlist IDs where the subscriber count is at least 100. The subscriber data is in playlist_subscribers.

Step 2 — Identify the approach

A set comprehension with no output expression: $\{p : S \;|\; P\}$ collects all $p$ from $S$ satisfying $P$. Here the output defaults to $p$ itself (the playlist ID).

  • Source: dom playlist_subscribers
  • Filter: $\#$(playlist_subscribers p) $\geq$ 100

Step 3 — Plan the components

  1. Iterate: draw $p$ from dom playlist_subscribers — the set of playlist IDs that have subscribers.
  2. Count: $\#$(playlist_subscribers p) gives the subscriber count for playlist $p$.
  3. Filter: keep only those where the count $\geq$ 100.

Solution

Click to reveal

Build Step 1 — Start the comprehension: iterate over playlist IDs

SymbolPlain EnglishWhat it does here
$\{\ \}$set braces — enclose a set comprehensionwraps the entire expression
$p$bound variable — represents each playlist ID in turnstands for each playlist ID as we iterate
$:$type declaration — "drawn from"says $p$ comes from the source set
domdomain — the set of all input values of a functiongives the set of playlist IDs with subscriber data

Open the braces, declare p : dom playlist_subscribers, leaving the filter as a placeholder:

$$\{p : \text{dom}\; playlist\_subscribers \;|\; \ldots\}$$

Build Step 2 — Count subscribers: $\#$(playlist_subscribers p)

SymbolPlain EnglishWhat it does here
playlist_subscribers $p$function application — applies the function to playlist $p$gives the set of users subscribed to playlist $p$
#cardinality — the number of elements in a setcounts how many users are in that set

playlist_subscribers $p$ gives the set of users subscribed to playlist $p$. $\#$ counts them:

$$\#(playlist\_subscribers\; p) = \text{number of subscribers of playlist } p$$

Build Step 3 — Add the filter: $\geq$ 100

SymbolPlain EnglishWhat it does here
$|$filter bar — restricts which elements are consideredintroduces the condition
$\geq$greater than or equal — left side is at least as large as right sidekeeps only playlists with 100+ subscribers

After $|$, require the count to be at least 100:

$$\{p : \text{dom}\; playlist\_subscribers \;|\; \#(playlist\_subscribers\; p) \geq 100\}$$

Since there is no output expression after $\bullet$, the output defaults to $p$ itself — the playlist ID.

Complete answer

$$B == \{p : \text{dom}\; playlist\_subscribers \;|\; \#(playlist\_subscribers\; p) \geq 100\}$$

"The set of all playlist IDs p in the domain of playlist_subscribers such that the cardinality of the subscriber set is at least 100."

Part (c) — The User Who Loves the Most Songs

Task: Find the unique user who loves strictly more songs than any other user. Assume this user is unique.

Vocab for Part (c)

TermDefinition
Mu-expression ($\mu$)$(\mu\, x : S \mid P)$ — "the unique $x$ in $S$ satisfying $P$." Only defined when exactly one such $x$ exists.
Universal quantifier ($\forall$)"For all." Asserts a property holds for every element in a set.
Cardinality (#)The number of elements in a finite set. $\#\{a, b, c\} = 3$.

Symbols for Part (c)

SymbolNamePlain English
$\mu$mu (unique selector)"the unique element satisfying..."
domdomainthe set of all input values of a function
lovedstate variablemaps each user to the set of songs they love
∀universal quantifier"for all"
≠not equaltwo values are different
•spotseparates constraint from expression in a quantifier
#cardinalitythe number of elements in a set
>strictly greater thanleft side exceeds right side

Step 1 — Understand the task

We need a mu-expression that finds the unique user who loves more songs than every other user. The problem tells us to assume uniqueness.

Precondition. The problem says "assume this user is unique." Without uniqueness, the $\mu$-expression would be undefined. We assume there is no tie for the top spot.

Step 2 — Identify the approach

A $\mu$-expression has the form ($\mu$ x : S | P). We need:

  • Source $S$: dom loved — users who love at least one song
  • Predicate $P$: for every other user, $u$ loves strictly more songs — use $\forall$ with $\neq$ and $>$

Step 3 — Plan the components

  1. $\mu$ skeleton: draw $u$ from dom loved.
  2. Predicate: $\forall$ $v$ : dom loved | $u$ $\neq$ $v$ $\bullet$ $\#$(loved $u$) $>$ $\#$(loved $v$).
  3. Assemble into a complete expression.

Solution

Build Step 1 — Start with the $\mu$ skeleton

SymbolPlain EnglishWhat it does here
$\mu$mu — "the unique element satisfying..."selects the single user who loves the most songs
$u$bound variable — represents a candidate userstands for the user we are testing
$:$type declaration — "drawn from"says $u$ comes from the source set
domdomain — the set of all input values of a functiongives the set of users who love at least one song
lovedstate variable — maps each user to the set of songs they loveprovides the source set via its domain

Open the $\mu$-expression, declare u : dom loved, leaving the predicate as a placeholder:

$$(\mu\, u : \text{dom}\; loved \;|\; \ldots)$$

Build Step 2 — Fill in the "most songs" predicate

SymbolPlain EnglishWhat it does here
∀universal quantifier — "for all"condition must hold for every other user
$v$bound variable — represents each comparison userthe user we compare $u$ against
$|$filter bar — restricts which elements are consideredintroduces the condition $u \neq v$
≠not equal — two values are differentexcludes $u$ itself from comparison
•spot — separates constraint from expressionseparates the filter from the condition
#cardinality — the number of elements in a setcounts how many songs each user loves
>strictly greater than — left side exceeds right side$u$'s count must beat every other user's

The predicate says "$u$ loves more songs than every other user." Use ∀ over dom loved, filter to exclude $u$ itself with ≠, then require #(loved $u$) > #(loved $v$):

$$(\mu\, u : \text{dom}\; loved \;|\; (\forall\, v : \text{dom}\; loved \;|\; u \neq v \bullet \#(loved\; u) > \#(loved\; v)))$$

Build Step 3 — Complete expression

The expression is now complete. The $\mu$ selects the unique $u$ from dom loved such that for every other user $v$, $u$ loves strictly more songs than $v$.

Let us trace a concrete example. Suppose:

  • $loved = \{u_1 \mapsto \{s_1, s_2, s_3\},\; u_2 \mapsto \{s_1, s_4\},\; u_3 \mapsto \{s_5\}\}$

Then $\#(loved\; u_1) = 3$, $\#(loved\; u_2) = 2$, $\#(loved\; u_3) = 1$.

  • $u = u_1$: Is $3 > 2$? Yes. Is $3 > 1$? Yes. Satisfies the predicate.
  • $u = u_2$: Is $2 > 3$? No. Fails.
  • $u = u_3$: Is $1 > 3$? No. Fails.

Therefore $C = u_1$.

Complete answer

$$C == (\mu\, u : \text{dom}\; loved \;|\; (\forall\, v : \text{dom}\; loved \;|\; u \neq v \bullet \#(loved\; u) > \#(loved\; v)))$$

Assumes a unique maximum exists. If two users tie for most loved songs, $\mu$ is undefined.

Part (d) — The Song Loved by the Most Users

Task: Find the unique song that is loved by strictly more users than any other song. Assume this song is unique.

Vocab for Part (d)

TermDefinition
Mu-expression ($\mu$)$(\mu\, x : S \mid P)$ — "the unique $x$ in $S$ satisfying $P$." Only defined when exactly one such $x$ exists.
Nested comprehensionA set comprehension used inside another expression — here, inside $\#\{\ldots\}$ to count elements satisfying a condition.
Cardinality (#)The number of elements in a finite set. $\#\{a, b, c\} = 3$.

Symbols for Part (d)

SymbolNamePlain English
$\mu$mu (unique selector)"the unique element satisfying..."
songsstate variablethe full set of all songs on the platform
∀universal quantifier"for all"
≠not equaltwo values are different
•spotseparates constraint from expression in a quantifier
#cardinalitythe number of elements in a set
>strictly greater thanleft side exceeds right side
$\{\ |\ \}$set comprehensioniterate over a source, keep elements satisfying a predicate
∈element of"belongs to" — a value is a member of a set
lovedstate variablemaps each user to the set of songs they love

Step 1 — Understand the task

We need a mu-expression that finds the unique song loved by more users than any other song. Unlike Part (c), loved is indexed by user, not by song. So counting "how many users love song $s$" requires a nested comprehension.

Key difficulty. In Part (c), counting was direct: #(loved $u$) gives the number of songs user $u$ loves. But here, loved is keyed by user, not by song. To count "how many users love song $s$", we must scan all users via a nested comprehension.

Step 2 — Identify the approach

Same "unique maximum" pattern as Part (c), but with a different counting mechanism:

  • Source: $s$ from songs
  • Counting: #$\{u : UserId \;|\; s$ ∈ loved $u\}$ counts users who love song $s$
  • Predicate: ∀ with ≠ and >

Step 3 — Plan the components

  1. $\mu$ skeleton: draw $s$ from songs.
  2. Counting expression: #$\{u : UserId \;|\; s$ ∈ loved $u\}$ — the nested comprehension.
  3. Comparison predicate: ∀ $t$ : songs | $s$ ≠ $t$ • count($s$) > count($t$).
  4. Assemble into a complete expression.

Solution

Build Step 1 — Start with the $\mu$ skeleton

SymbolPlain EnglishWhat it does here
$\mu$mu — "the unique element satisfying..."selects the single song loved by the most users
$s$bound variable — represents a candidate songstands for the song we are testing
$:$type declaration — "drawn from"says $s$ comes from the source set
songsstate variable — the full set of all songs on the platformprovides the source set

Open the $\mu$-expression, declare s : songs, leaving the predicate as a placeholder:

$$(\mu\, s : songs \;|\; \ldots)$$

Note: we draw from songs (not dom loved), because loved is keyed by users, not songs.

Build Step 2 — Build the counting expression: $\#${$u$ : UserId | $s$ ∈ loved $u$}

SymbolPlain EnglishWhat it does here
$\{\ |\ \}$set comprehension — iterate, filter, collectbuilds the set of users who love song $s$
$u$bound variable — represents each useriterates over all users
∈element of — "belongs to"checks whether song $s$ is in user $u$'s love-set
#cardinality — the number of elements in a setcounts the resulting set of users

We construct a nested comprehension that collects all users who love a given song $s$:

$$\{u : UserId \;|\; s \in loved\; u\}$$

This reads: "the set of all users $u$ such that song $s$ belongs to $u$'s love-set." The cardinality #$\{u : UserId \;|\; s \in loved\; u\}$ is then the number of users who love song $s$.

Build Step 3 — Fill in the comparison predicate: ∀ $t$, ≠, >

SymbolPlain EnglishWhat it does here
∀universal quantifier — "for all"condition must hold for every other song
$t$bound variable — represents each comparison songthe song we compare $s$ against
≠not equal — two values are differentexcludes $s$ itself from comparison
•spot — separates constraint from expressionseparates the filter from the condition
>strictly greater than — left side exceeds right side$s$'s love-count must beat every other song's

For each candidate song $s$, compare its "love count" against every other song $t$. Use ∀ over songs, filter with ≠, then require >:

$$\forall\, t : songs \;|\; s \neq t \bullet \#\{u : UserId \;|\; s \in loved\; u\} > \#\{u : UserId \;|\; t \in loved\; u\}$$

Build Step 4 — Complete expression

Wrap the predicate inside the $\mu$:

$$(\mu\, s : songs \;|\; (\forall\, t : songs \;|\; s \neq t \bullet \#\{u : UserId \;|\; s \in loved\; u\} > \#\{u : UserId \;|\; t \in loved\; u\}))$$

Let us trace a concrete example. Suppose:

  • $songs = \{s_1, s_2, s_3\}$
  • $loved = \{u_1 \mapsto \{s_1, s_2\},\; u_2 \mapsto \{s_1, s_3\},\; u_3 \mapsto \{s_1\}\}$

Count the love-fans for each song:

  • $s_1$: Loved by $u_1, u_2, u_3$ — count = 3
  • $s_2$: Loved by $u_1$ — count = 1
  • $s_3$: Loved by $u_2$ — count = 1

Checking $s = s_1$: Is $3 > 1$ (vs $s_2$)? Yes. Is $3 > 1$ (vs $s_3$)? Yes. So $s_1$ satisfies.

Therefore $D = s_1$.

Complete answer

$$D == (\mu\, s : songs \;|\; (\forall\, t : songs \;|\; s \neq t \bullet \#\{u : UserId \;|\; s \in loved\; u\} > \#\{u : UserId \;|\; t \in loved\; u\}))$$

Assumes a unique maximum exists. If two songs tie for most loved, $\mu$ is undefined.
