Exercise 40 — TV Recording System

Modelling exercise — building a formal specification step by step.

The Scenario

A TV recording system records television programmes to a hard-drive.

  • The hard-drive can store up to 200 hours of programming.
  • Each programme may be at most 6 hours in length.

The viewer sees a menu listing each programme's title, programme length, and whether or not it has been viewed.

Given Types

DeclarationWhat it isNotes
$[\mathit{Title}]$Basic typeAbstract set of all possible titles
$\mathit{Length} == \mathbb{N}$AbbreviationA natural number, in minutes
$\mathit{Viewed} ::= yes \mid no$Free typeExactly two values

Key detail: lengths are in minutes. "200 hours" = $12\,000$ minutes. "6 hours" = $360$ minutes.

Each programme is a triple of type $\mathit{Title} \times \mathit{Length} \times \mathit{Viewed}$, and the hard-drive stores a sequence of these:

$$hd : \text{seq}(\mathit{Title} \times \mathit{Length} \times \mathit{Viewed})$$

Part (a) — Axiomatic Definition with Constraints

Task: Complete the axiomatic definition of $hd$ with constraints for the 200-hour capacity limit and the 6-hour per-programme limit. You may assume $\mathit{cumulative\_total}$ exists (defined in part d).

Vocab for Part (a)

TermDefinition

PredicateA statement that is either true or false.
InvariantA condition that must be satisfied in every valid state of the system.
ConstraintA restriction on the valid values a variable may take.

Symbols for Part (a)

SymbolNamePlain English

Step 1 — Understand the task

We need to complete an axiomatic definition. The declaration ($hd$ and its type) goes above the line. Below the line, we write predicates — invariants that must always hold.

Step 2 — Identify the approach

We have two English constraints. Each becomes one predicate below the line.

  1. "the hard-drive can store up to 200 hours" → bound the total length
  2. "each programme may be at most 6 hours" → bound each individual length

Step 3 — Plan the components

Constraint 1: sum of all lengths ≤ 12,000 minutes (200 × 60). Use cumulative_total to compute the sum.

Constraint 2: for every programme, its length ≤ 360 minutes (6 × 60). Use ∀ over ran hd to reach each programme, and p.2 to extract the length.

Why ran hd, not dom hd? A sequence is a function from indices {1, 2, ..., n} to values. The domain is the indices. The range is the actual programme triples.

Example: $\mathit{viewed}\;\langle(t_1,3,yes),(t_2,4,yes),(t_1,5,no)\rangle = \langle(t_1,3,yes),(t_2,4,yes)\rangle$

Step 1 — Understand the task

We need two functions that filter a sequence of programmes. The function viewed keeps only entries whose third component is $yes$. The function not_viewed keeps only entries whose third component is $no$. Both return a sequence of the same type.

Step 2 — Identify the approach

Two clean ways to filter a sequence in Z:

  1. Recursive definition: process one element at a time (base case + recursive case).
  2. Sequence restriction: use ↾ to keep only elements from a given set.

We show recursive for $\mathit{viewed}$ and restriction for $\mathit{not\_viewed}$.

Recursive viewed: the base case returns ⟨⟩. The recursive case checks x.3: if $yes$, keep; otherwise skip.

Part (d) — Recursive cumulative_total

Example: $\mathit{cumulative\_total}\;\langle(t_1,3,yes),(t_2,4,yes),(t_1,5,no)\rangle = 12$

Every recursive function on sequences follows the same skeleton:

Step 1 — Understand the task

We need a function that takes a sequence of programme triples and returns the same elements rearranged so that lengths are in descending order. This is a specification — we describe what the result must look like, not how to sort.

Step 2 — Identify the approach

A specification-style sort uses two properties:

  1. Permutation: the output contains exactly the same elements as the input (checked via bag equality using items).
  2. Descending order: for any two positions i < j, the length at position i is ≥ the length at position j.

Step 3 — Plan the components

Permutation: items(s x) = items(x) — same bag means same elements with same multiplicities.

Ordering: ∀ i, j : dom(s x) • i < j ⇒ ((s x) i).2 ≥ ((s x) j).2. Use ≥ (not >) to allow ties.

Combine: join both properties with ∧.
