Exercise 43 — Cinema Booking System

Modelling exercise — building a formal specification step by step.

The Scenario

An online booking system for a group of cinemas reserves seats for individual customers. Each performance is uniquely identified by a 4-tuple: (Cinema, Film, Date, Time). The system records which customers have booked which seats for which performances.

The booking information is captured by an axiomatic definition:

$$bookings : \text{Cinema} \times \text{Film} \times \text{Date} \times \text{Time} \pfun \mathbb{P}(\text{Customer} \times \mathbb{N} \times \mathbb{N})$$

Each performance maps to a set of booking triples $(c, m, n)$ where:

  • $c \in \text{Customer}$ — the customer who made the booking
  • $m \in \mathbb{N}$ — first seat number of the reserved block (lower bound)
  • $n \in \mathbb{N}$ — last seat number of the reserved block (upper bound)
  • The reserved seats are the integer range $m \,..\, n$
  • If a person bought one ticket, then $m = n$

You may assume the existence of a function $\text{max} : \text{Cinema} \to \mathbb{N}$ that maps a cinema to the number of seats available in that cinema. The range of seats for each cinema $c$ runs from $1$ to $\text{max}\,c$.

Part (a) — Constraints on bookings

We must express four constraints in predicate logic that ensure the booking system is well-formed.

(i) No seat sold to more than one customer

Task: Write a predicate ensuring that for any single performance, no two distinct booking triples reserve overlapping seats.

Step 1 — Understand the task

We need a predicate that prevents double-booking of seats. For each performance $i$, the set bookings $i$ contains all booking triples for that showing. No seat number should appear in two different bookings.

Step 2 — Identify the approach

We use nested quantification: an outer ∀ iterates over performances, and an inner ∀ iterates over pairs of distinct bookings within each performance. For each pair, we require their seat ranges to be disjoint.

Step 3 — Plan the components

  • Outer loop: ∀ i : dom bookings — iterate over every performance.
  • Inner loop: ∀ x, y : bookings i | x ≠ y — take any two different bookings.
  • Condition: (x.2..x.3) ∩ (y.2..y.3) = ∅ — their seat ranges must not overlap.

(ii) All seats within valid range

Task: Write a predicate ensuring every booked seat number falls within the range $1 \,..\, \text{max}\,c$ where $c$ is the cinema of the performance.

Step 1 — Understand the task

We need a predicate that ensures every booked seat is a real seat in the cinema. For a performance $i$, the cinema is $i.1$ and the valid seat numbers are $1 \,..\, \text{max}(i.1)$. Both the lower and upper bounds of each block booking must lie in this range.

Step 2 — Identify the approach

Again we use nested quantification: an outer ∀ over performances and an inner ∀ over booking triples within each performance. For each booking, we check that the seat endpoints fall within the valid range using ⊆.

Step 3 — Plan the components

  • Outer loop: ∀ i : dom bookings — iterate over every performance.
  • Inner loop: ∀ x : bookings i — iterate over every booking triple.
  • Condition: {x.2, x.3} ⊆ 1..max(i.1) — both endpoints lie in valid range.

(iv) No two reserved blocks overlap

Task: State whether a separate constraint is needed to prevent two reserved blocks from overlapping, or whether this is already captured by an earlier constraint.

Vocab for Part (a)(iv)

Symbols for Part (a)(iv)

SymbolNamePlain English

Step 1 — Understand the task

"No two reserved blocks overlap" means that for distinct bookings at the same performance, their seat ranges share no seat number. This is a disjointness requirement.

Step 2 — Identify the approach

Compare this requirement with constraint (i). Constraint (i) says: for any two different bookings $x \ne y$ at the same performance, $(x.2 \,..\, x.3) \cap (y.2 \,..\, y.3) = \emptyset$. This is exactly the "no overlapping blocks" requirement.

Step 3 — Plan the components

No new formula is needed. The pairwise disjointness of seat ranges in constraint (i) already prevents any two reserved blocks from sharing a seat.

Part (b) — Set Comprehensions

Step 1 — Understand the task

We need a set comprehension that pairs each film with a (cinema, day-count) pair. The day-count is the number of distinct dates on which that film was shown at that cinema. This requires a nested set comprehension to count the dates.

Step 2 — Identify the approach

Two layers of work:

  1. Outer: iterate over all (Cinema, Film) combinations that actually appear in the bookings.
  2. Inner: for each such pair, count the distinct dates using cardinality of a nested set comprehension.

We use an existential quantifier in the filter to ensure we only output (cinema, film) pairs that genuinely appear in the bookings.

Step 3 — Plan the components

  • Declaration: c : Cinema; f : Film

  • Output: (f, (c, count))
