Exercise 51 — Streamify: Song Length and Popularity

Software Engineering Mathematics — Z Notation Modelling (Part 4 of a 5-exercise series)

The Scenario

Two further functions are introduced to the Streamify platform. The first, length, maps a given song identifier to its length in seconds. The second, popularity, assigns a popularity score to each song identifier.

$$length : SongId \pfun \mathbb{N}$$ $$popularity : SongId \pfun \mathbb{N}$$

Context from Exercises 48–50

From the combined Streamify model, we have:

  • $songs : \mathbb{F}\; SongId$ — the finite set of all songs on the platform
  • $users : \mathbb{F}\; UserId$ — the finite set of all registered users
  • $playlists : PlaylistId \pfun Playlist$ where $Playlist == \text{iseq}_1\; SongId$ — each playlist is a non-empty injective sequence of songs
  • $playlist\_owner : PlaylistId \pfun UserId$ — who owns each playlist
  • $playlist\_subscribers : PlaylistId \pfun \mathbb{F}_1\; UserId$ — who subscribes to each playlist
  • $hated : UserId \pfun \mathbb{F}\; SongId$ — songs each user hates
  • $loved : UserId \pfun \mathbb{F}\; SongId$ — songs each user loves

New Declarations

Two new partial functions map songs to natural numbers:

$$length : SongId \pfun \mathbb{N}$$ $$popularity : SongId \pfun \mathbb{N}$$

$length$ gives the duration of a song in seconds. $popularity$ computes a score based on user reactions and playlist appearances.

Part (a) — Complete the Popularity Definition

Task: Complete the definition so that:

  1. Songs appearing in length and popularity must appear in songs.
  2. The popularity of a song is: (number who love it $-$ number who hate it), capped at 0 if negative, multiplied by the number of playlists in which it appears.

Example 1: a song liked by 6, hated by 3, in 10 playlists $\Rightarrow$ $(6 - 3) \times 10 = 30$.

Example 2: a song liked by 3, hated by 6, in 10 playlists $\Rightarrow$ $0 \times 10 = 0$ (capped at 0).

Love-hate scoreThe net positive sentiment for a song: max(lovers − haters, 0).

Step 1 — Understand the task

We need to complete the definition of length and popularity by adding constraints below the line. There are two aspects: (1) domain constraints that tie these functions to songs, and (2) a formula for popularity that combines user sentiment with playlist exposure. Since the formula is complex, we define two helper functions first.

Step 2 — Identify the approach

The popularity formula has two independent pieces:

  • Net sentiment — how many more people love a song than hate it, capped at zero to stay in $\mathbb{N}$.
  • Exposure — how many playlists feature the song.

We define a love_hate_score helper for sentiment and a playlist_count helper for exposure. Then the main definition multiplies them.

Step 3 — Plan the components

  1. Domain constraints: dom length ⊆ songs and dom popularity ⊆ songs.
  2. Helper 1 (love_hate_score): two conditional clauses using ∀ and ⇒ — one for the non-negative case, one for the negative case.
  3. Helper 2 (playlist_count): count playlists containing a song using # and a set comprehension over dom playlists.
  4. Main formula: popularity $i$ = love_hate_score $i$ × playlist_count $i$.

Build Step 1 — Domain constraints: dom ⊆ songs

SymbolPlain EnglishWhat it does here
domdomain — the set of inputs on which a function is definedextracts the set of song IDs for which length / popularity is defined
⊆subset — every element of the left set is also in the right setensures only known songs have length or popularity values

Both functions must only be defined for songs that actually exist in the system. Use dom on each function and assert ⊆ songs:

$$\text{dom}\; length \;\subseteq\; songs$$ $$\text{dom}\; popularity \;\subseteq\; songs$$

Build Step 2 — Define love_hate_score — Case 1: lovers ≥ haters

SymbolPlain EnglishWhat it does here
∀universal quantifier — "for all"the clause must hold for every song
$i$bound variable — represents each song ID in turnstands for each song as we iterate
songsstate variable — the finite set of all songsthe set we iterate over
•spot — separates the bound variable from the bodyconnects "for all $i$" to the condition
#cardinality — the number of elements in a setcounts users who love or hate the song
≥greater than or equaltests whether lovers outnumber haters
⇒implies — "if ... then ..."when the condition holds, the score is the difference
−subtraction — arithmetic differencecomputes lovers minus haters

For every song i in songs, if the number of users who love it is at least as large as the number who hate it, the score is the difference. Use # with a set comprehension to count lovers and haters:

$$\forall\; i : songs \;\bullet$$ $$\quad \#\{u : UserId \mid i \in loved\;u\} \;\geq\; \#\{u : UserId \mid i \in hated\;u\}$$ $$\quad \Rightarrow\; love\_hate\_score\;i \;=\; \#\{u : UserId \mid i \in loved\;u\} - \#\{u : UserId \mid i \in hated\;u\}$$

This reads: "For every song $i$, if the lover count ≥ the hater count, ⇒ the score is lovers − haters."

Build Step 3 — Define love_hate_score — Case 2: lovers < haters (capped at zero)

SymbolPlain EnglishWhat it does here
∧conjunction — "and"joins this case to the previous one
<strictly less than — left side is smallertests whether haters outnumber lovers
⇒implies — "if ... then ..."when more people hate it, the score is clamped to 0

The second clause is joined by ∧. When more users hate the song than love it, the score is capped at 0 instead of going negative:

$$\land$$ $$\forall\; i : songs \;\bullet$$ $$\quad \#\{u : UserId \mid i \in loved\;u\} \;<\; \#\{u : UserId \mid i \in hated\;u\}$$ $$\quad \Rightarrow\; love\_hate\_score\;i \;=\; 0$$

These two cases are exhaustive ($\geq$ or $<$ cover all possibilities) and mutually exclusive, so every song gets exactly one value.

Build Step 4 — Worked examples for love_hate_score

Example 1: Song $i$ is loved by 6 users, hated by 3.

  1. Count lovers: $\#\{u : UserId \mid i \in loved\;u\} = 6$
  2. Count haters: $\#\{u : UserId \mid i \in hated\;u\} = 3$
  3. Compare: $6 \geq 3$ — Case 1 applies.
  4. love_hate_score: $love\_hate\_score\;i = 6 - 3 = 3$

Example 2: Song $j$ is loved by 3 users, hated by 6.

  1. Count lovers: $\#\{u : UserId \mid j \in loved\;u\} = 3$
  2. Count haters: $\#\{u : UserId \mid j \in hated\;u\} = 6$
  3. Compare: $3 < 6$ — Case 2 applies.
  4. love_hate_score: $love\_hate\_score\;j = 0$ (capped at zero).

Build Step 5 — Define playlist_count using # and a nested set comprehension

SymbolPlain EnglishWhat it does here
#cardinality — the number of elements in a setcounts the playlists that contain the song
domdomain — the set of inputs on which a function is definedgives the set of all playlist IDs
playlistsstate variable — maps each playlist ID to its sequence of songsthe function we iterate over
ranrange — the set of output values of a functionextracts the set of songs from a playlist sequence
∈element of — "belongs to"checks whether the song appears in a playlist

For each song i, count how many playlists contain it. A playlist $p$ contains song $i$ if $i \in \text{ran}(playlists\;p)$. Wrap that set comprehension in # to get the count:

$$\forall\; i : songs \;\bullet\; playlist\_count\;i \;=\; \#\{p : \text{dom}\;playlists \mid i \in \text{ran}(playlists\;p)\}$$

This reads: "For every song $i$, the playlist count is the number of playlist IDs $p$ whose range contains $i$."

Build Step 6 — Combine: popularity = love_hate_score × playlist_count

SymbolPlain EnglishWhat it does here
∀universal quantifier — "for all"the formula holds for every song
×multiplication — arithmetic productmultiplies sentiment by exposure

The final piece ties the two helpers together. For each song, popularity is the product of its love_hate_score and its playlist_count:

$$\forall\; i : songs \;\bullet\; popularity\;i \;=\; love\_hate\_score\;i \;\times\; playlist\_count\;i$$

Now we can complete the worked examples:

  • Example 1: $love\_hate\_score\;i = 3$, $playlist\_count\;i = 10$. So $popularity\;i = 3 \times 10 = 30$.
  • Example 2: $love\_hate\_score\;j = 0$, $playlist\_count\;j = 10$. So $popularity\;j = 0 \times 10 = 0$.

Even though song $j$ appears in many playlists, its popularity is zero because more people hate it than love it.

Complete answer

Helper 1: love_hate_score

$$love\_hate\_score : SongId \pfun \mathbb{N}$$
$$\forall\; i : songs \;\bullet$$ $$\quad \#\{u : UserId \mid i \in loved\;u\} \;\geq\; \#\{u : UserId \mid i \in hated\;u\}$$ $$\quad \Rightarrow\; love\_hate\_score\;i \;=\; \#\{u : UserId \mid i \in loved\;u\} - \#\{u : UserId \mid i \in hated\;u\}$$ $$\;\land$$ $$\forall\; i : songs \;\bullet$$ $$\quad \#\{u : UserId \mid i \in loved\;u\} \;<\; \#\{u : UserId \mid i \in hated\;u\}$$ $$\quad \Rightarrow\; love\_hate\_score\;i \;=\; 0$$

Helper 2: playlist_count

$$playlist\_count : SongId \pfun \mathbb{N}$$
$$\forall\; i : songs \;\bullet\; playlist\_count\;i \;=\; \#\{p : \text{dom}\;playlists \mid i \in \text{ran}(playlists\;p)\}$$

Main definition

$$length : SongId \pfun \mathbb{N}$$ $$popularity : SongId \pfun \mathbb{N}$$
$$\text{dom}\;length \;\subseteq\; songs$$ $$\text{dom}\;popularity \;\subseteq\; songs$$ $$\forall\; i : songs \;\bullet\; popularity\;i \;=\; love\_hate\_score\;i \;\times\; playlist\_count\;i$$

Part (b) — Most Popular Song (Axiomatic Definition)

Task: Define, via an axiomatic definition, the most popular song in songs. If no unique most popular song exists, return null_song.

Vocab for Part (b)

TermDefinition
Unique existence ($\exists_1$)There is exactly one element satisfying the predicate. True when precisely one witness exists, false otherwise.
Sentinel valueA fallback value used when no valid result exists. Here, null_song signals "no unique winner."
Case splitDefining a value differently depending on whether a condition holds. The cases must be exhaustive.

≠not equaltwo values are different
>strictly greater thanleft side exceeds right side
•spotseparates constraint from expression

Step 1 — Understand the task

We need an axiomatic definition that declares a constant most_popular of type $SongId$, with predicates below the line that determine its value. There are two cases: either there is a unique song that beats all others (use $\mu$ to extract it), or there is not (return null_song).

Step 2 — Identify the approach

A case split using two implications joined by ∧:

  • Case 1: $\exists_1$ confirms a unique winner exists — use $\mu$ to pick it out.
  • Case 2: $\lnot$$\exists_1$ — no unique winner (tie or empty) — use null_song.

Together these cover all possibilities, making the definition total.

Step 3 — Plan the components

  1. Declare: most_popular $: SongId$
  2. Uniqueness test: $\exists_1$ $i : songs$ such that ∀ $j : songs \mid i \neq j$ • popularity $i$ > popularity $j$
  3. Case 1: uniqueness holds ⇒ use μ
  4. Case 2: uniqueness fails ⇒ null_song

Build Step 1 — Declare most_popular

SymbolPlain EnglishWhat it does here

$:$type declaration — "is of type"declares the type of the constant

Above the line, declare the constant:

$$most\_popular : SongId$$

Build Step 2 — Case 1: unique winner exists — use $\exists_1$ and $\mu$

SymbolPlain EnglishWhat it does here
∃ (with subscript 1)unique existence — "there exists exactly one"tests whether exactly one song beats all others
∀universal quantifier — "for all"requires the winner to beat every other song
≠not equal — two values are differentexcludes the candidate from comparison with itself
>strictly greater thanthe winner's popularity must strictly exceed every other song's
⇒implies — "if ... then ..."if the unique winner exists, then most_popular equals it
μmu — unique selectorextracts the unique song satisfying the predicate

The first implication: if there exists exactly one song whose popularity strictly exceeds every other song's, then most_popular is that unique song (selected via $\mu$):

$$(\exists_1\; i : songs \;\bullet\; (\forall\; j : songs \mid i \neq j \;\bullet\; popularity\;i > popularity\;j))$$ $$\;\Rightarrow\; most\_popular \;=\; (\mu\; i : songs \mid (\forall\; j : songs \mid i \neq j \;\bullet\; popularity\;i > popularity\;j))$$

Build Step 3 — Case 2: no unique winner — default to null_song

SymbolPlain EnglishWhat it does here
¬negation — "not"reverses the uniqueness test
⇒implies — "if ... then ..."when no unique winner exists, fall back to sentinel
null_songsentinel value — "no valid result"the fallback value

The second implication: if the uniqueness condition fails — either two songs tie for the top, or songs is empty — then most_popular is the sentinel:

$$\lnot(\exists_1\; i : songs \;\bullet\; (\forall\; j : songs \mid i \neq j \;\bullet\; popularity\;i > popularity\;j))$$ $$\;\Rightarrow\; most\_popular \;=\; null\_song$$

Build Step 4 — Combine with ∧

The two cases are joined by ∧. Since a statement either has a unique witness or it does not, exactly one implication has a true antecedent in any given state, making most_popular always defined.

Complete answer

$$most\_popular : SongId$$
$$(\exists_1\; i : songs \;\bullet\; (\forall\; j : songs \mid i \neq j \;\bullet\; popularity\;i > popularity\;j))$$ $$\;\Rightarrow\; most\_popular \;=\; (\mu\; i : songs \mid (\forall\; j : songs \mid i \neq j \;\bullet\; popularity\;i > popularity\;j))$$ $$\;\land$$ $$\lnot(\exists_1\; i : songs \;\bullet\; (\forall\; j : songs \mid i \neq j \;\bullet\; popularity\;i > popularity\;j))$$ $$\;\Rightarrow\; most\_popular \;=\; null\_song$$

Example: Suppose $popularity\;s_1 = 30$, $popularity\;s_2 = 12$, $popularity\;s_3 = 0$.

  • $s_1$ has $30 > 12$ and $30 > 0$ — it beats every other song.
  • No other song beats all others.
  • So $\exists_1$ is satisfied with witness $s_1$, and $most\_popular = s_1$.

Tie example: Suppose $popularity\;s_1 = 30$ and $popularity\;s_2 = 30$. Neither has strictly greater popularity than the other, so no song satisfies "beats all others." The $\exists_1$ fails, and $most\_popular = null\_song$.

Part (c) — Playlists Containing the Most Popular Song

Task: Define, via a set comprehension, the set of playlists that feature most_popular. Assume null_song appears in no playlists.

Step 1 — Understand the task

We need an abbreviation for the set of all playlist IDs whose content includes most_popular. Since null_song does not appear in any playlist, this set will be empty when there is no unique winner — a sensible default.

Step 2 — Identify the approach

A set comprehension with no output expression: $\{i : S \mid P\}$ collects all $i$ from $S$ satisfying $P$. The output defaults to $i$ itself (the playlist ID).

  • Source: dom playlists
  • Filter: most_popular ∈ ran(playlists $i$)

Step 3 — Plan the components

  1. Iterate over dom playlists — the set of all playlist IDs.
  2. Check whether most_popular ∈ ran(playlists $i$) — whether the most popular song appears in that playlist's sequence.
  3. Wrap in an abbreviation using $==$.

Build Step 2 — Check most_popular ∈ ran(playlists $i$)

SymbolPlain EnglishWhat it does here

After the filter bar, check that most_popular belongs to the range of the playlist sequence for playlist $i$:

$$\{i : \text{dom}\;playlists \mid most\_popular \in \text{ran}(playlists\;i)\}$$

Build Step 3 — Complete abbreviation

Give the expression a name using $==$:

$$playlists\_containing\_most\_popular\_song \;==\; \{i : \text{dom}\;playlists \mid most\_popular \in \text{ran}(playlists\;i)\}$$

Complete answer

$$playlists\_containing\_most\_popular\_song \;==\; \{i : \text{dom}\;playlists \mid most\_popular \in \text{ran}(playlists\;i)\}$$

"The set of all playlist IDs i in the domain of playlists such that most_popular belongs to the range of that playlist."

Example: Suppose $most\_popular = s_1$ and:

  • $playlists\;p_1 = \langle s_1, s_3, s_5 \rangle$ — $\text{ran} = \{s_1, s_3, s_5\}$ — $s_1 \in \text{ran}$? Yes. Include $p_1$.
  • $playlists\;p_2 = \langle s_2, s_4 \rangle$ — $\text{ran} = \{s_2, s_4\}$ — $s_1 \in \text{ran}$? No. Exclude.
  • $playlists\;p_3 = \langle s_1, s_2 \rangle$ — $\text{ran} = \{s_1, s_2\}$ — $s_1 \in \text{ran}$? Yes. Include $p_3$.

Result: $playlists\_containing\_most\_popular\_song = \{p_1, p_3\}$.

Edge case: If $most\_popular = null\_song$ (no unique winner), this set will be empty — since $null\_song$ does not appear in any playlist — which is a sensible default.
