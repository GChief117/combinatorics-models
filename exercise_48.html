Exercise 48 — Streamify: Base System (Playlists)

Z Notation Modelling — Part 1 of a 5-exercise series

The Scenario

You have been asked to design (via a formal model) a prototype for a new online music platform called Streamify. This exercise establishes the base system: songs, users, playlists, ownership, and subscriptions.

All remaining questions (Exercises 49–52) will leverage the definitions built here.

Given Types

Three basic types representing atomic identifiers:

$$[PlaylistId,\; SongId,\; UserId]$$

A playlist is defined as a non-empty injective sequence of songs:

$$Playlist == \text{iseq}_1\; SongId$$
Unpacking this definition:
  • $\text{iseq}\; X$ is the set of all injective sequences over $X$ — sequences where no element appears twice.
  • The subscript $_1$ adds the constraint that the sequence must be non-empty (at least one element).
  • So a $Playlist$ is a finite sequence of $SongId$ values with no duplicate songs and at least one song.

System Declaration

The state of the Streamify base system is captured by five variables:

$playlists : PlaylistId \;{\pfun}\; Playlist$

The Task

Task: Complete the axiomatic definition of the Streamify system by adding predicates (below the line) that capture the following six constraints:
  1. Only songs that appear in $songs$ can appear in playlists.
  2. Only playlists that appear in $playlists$ can appear in $playlist\_owner$.
  3. Only users that appear in $users$ can appear in $playlist\_owner$.
  4. Only playlists that appear in $playlists$ can appear in $playlist\_subscribers$.
  5. Only users that appear in $users$ can appear in $playlist\_subscribers$.
  6. Every playlist's owner is, by default, a subscriber to that playlist.

Part (a) — Only songs in songs can appear in playlists

Task: Write a predicate ensuring that every song appearing in any playlist must be a member of the $songs$ set.

Vocab for Part (a)

SubsetEvery element of A is also in B.

Symbols for Part (a)

ranrangethe set of outputs that a function actually produces
•spotseparates the bound variable from the body in a quantified expression
⊆subsetevery element of the left set is also in the right set

Step 1 — Understand the task

We need a predicate that enforces referential integrity for songs. Every song that appears inside any playlist must exist in the global $songs$ set. This prevents playlists from referencing phantom or deleted tracks.

Step 2 — Identify the approach

Each playlist is an injective sequence of songs — a function from positions to $SongId$ values. So $playlists$ maps to sequences, not directly to individual songs. We cannot simply write ran playlists ⊆ songs because that would give us a set of sequences, not a set of songs.

Instead, we must iterate over each playlist and extract the set of songs from that individual sequence using ran.

Step 3 — Plan the components

  • Use ∀ to iterate over every playlist ID in dom playlists.
  • For each playlist ID $i$, apply playlists to get the sequence: $playlists\; i$.
  • Extract the set of songs in that sequence using ran: $\text{ran}(playlists\; i)$.
  • Assert that set is a subset of songs: $\text{ran}(playlists\; i) \subseteq songs$.

"For every playlist ID $i$ in the system, the set of songs in that playlist (obtained via ran) must be a subset of songs."

Part (b) — Only playlists in playlists can appear in playlist_owner

Task: Write a predicate ensuring that only playlist IDs that exist in $playlists$ can appear as keys in $playlist\_owner$.

Vocab for Part (b)

Step 1 — Understand the task

We need a domain constraint that prevents $playlist\_owner$ from assigning an owner to a playlist that does not exist. This is referential integrity on the key side of the function.

Step 2 — Identify the approach

Both $playlist\_owner$ and $playlists$ are partial functions keyed by $PlaylistId$. We need every key of $playlist\_owner$ to also be a key of $playlists$. This is a direct domain constraint: dom of one ⊆ dom of the other.

Step 3 — Plan the components

  • Extract the keys of $playlist\_owner$ using dom.
  • Extract the keys of $playlists$ using dom.
  • Assert the first is a subset of the second.

Solution

"The set of playlist IDs that have owners must be a subset of the set of playlist IDs that actually exist."

Why ⊆ and not =? We use ⊆ deliberately: it allows for the possibility that some playlists might not yet have an owner assigned (e.g., during creation). If we wanted to enforce that every playlist has an owner, we would write dom playlist_owner = dom playlists.

Step 1 — Understand the task

We need referential integrity for the subscriber values. Every user listed as a subscriber must exist in $users$. This is conceptually like Part (c), but the structure of $playlist\_subscribers$ is different.

Step 2 — Identify the approach

We must use ∀ to iterate over each playlist, unwrap the set-valued function, and check each subscriber set individually.

Part (f) — Every playlist's owner is a subscriber to that playlist

Task: Write a predicate ensuring that for every playlist, the owner is automatically a member of the subscriber set.

Vocab for Part (f)

TermDefinition
MembershipA single element belongs to a set (∈ vs ⊆).
Referential integrityEnsuring that references to entities (songs, users, playlists) point to things that actually exist.
Non-empty finite set (F₁)A finite set with at least one element.

Symbols for Part (f)

SymbolNamePlain English
∀universal quantifier"for all"
domdomainthe set of inputs on which a function is defined
•spotseparates the bound variable from the body in a quantified expression
∈element ofthe value on the left is a member of the set on the right

Step 1 — Understand the task

We need a business rule: when you create a playlist, you automatically follow it. The owner must always appear in the subscriber set. This is a membership constraint, not a subset constraint, because the owner is a single user, not a set.

Step 2 — Identify the approach

For each playlist, $playlist\_owner\; i$ yields a single $UserId$, while $playlist\_subscribers\; i$ yields a set of $UserId$ values. We need to assert that the single value belongs to the set. This calls for ∈, not ⊆.

Key distinction: use ∈ when the left side is an element; use ⊆ when the left side is a set.

Step 3 — Plan the components

  • Use ∀ to iterate over every playlist ID in dom playlists.
  • For each playlist ID $i$, get the owner: $playlist\_owner\; i$ (a single $UserId$).
  • Get the subscriber set: $playlist\_subscribers\; i$ (a set of $UserId$).
  • Assert membership: the owner ∈ the subscriber set.

SymbolPlain EnglishWhat it does here

$$\forall\, i : \text{dom}\; playlists \bullet (playlist\_owner\; i) \in playlist\_subscribers\; i$$

Complete answer

$$\forall\, i : \text{dom}\; playlists \bullet (playlist\_owner\; i) \in playlist\_subscribers\; i$$

"For every playlist, the owner is a member of the subscriber set."

Why ∈ and not ⊆? The owner is a single $UserId$, not a set. We use ∈ to test membership of an element in a set, and ⊆ to test containment of one set within another.

Why this constraint matters: It captures the business rule that creating a playlist automatically subscribes you to it. It also explains why $playlist\_subscribers$ maps to $\mathbb{F}_1$ (non-empty finite set) rather than $\mathbb{F}$: since the owner is always a subscriber, every playlist has at least one subscriber.

Complete System — Full Axiomatic Definition

Assembling all five declarations and six constraints into a single schema:

Streamify
$songs : \mathbb{F}\; SongId$
$users : \mathbb{F}\; UserId$
$playlists : PlaylistId \;{\pfun}\; Playlist$
$playlist\_owner : PlaylistId \;{\pfun}\; UserId$
$playlist\_subscribers : PlaylistId \;{\pfun}\; \mathbb{F}_1\; UserId$
$\forall\, i : \text{dom}\; playlists \bullet \text{ran}(playlists\; i) \subseteq songs$    ...(a)
$\text{dom}\; playlist\_owner \subseteq \text{dom}\; playlists$    ...(b)
$\text{ran}\; playlist\_owner \subseteq users$    ...(c)
$\text{dom}\; playlist\_subscribers \subseteq \text{dom}\; playlists$    ...(d)
$\forall\, i : \text{dom}\; playlist\_subscribers \bullet playlist\_subscribers\; i \subseteq users$    ...(e)
$\forall\, i : \text{dom}\; playlists \bullet (playlist\_owner\; i) \in playlist\_subscribers\; i$    ...(f)

Reading the schema: Above the line are the five variable declarations — the state space. Below the line are the six referential integrity constraints — the invariants. Every valid state of the Streamify system must satisfy all six predicates simultaneously.
