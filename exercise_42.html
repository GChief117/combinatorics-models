Exercise 42 — Supermarket Checkouts

Modelling exercise — building a formal specification step by step.

The Scenario

We model the checkout component of a supermarket: the collection of checkout points at which customers queue to pay. At each checkout, we track the sequence of people queueing to be served.

  • A supermarket is modelled as a sequence of checkouts.
  • A checkout is modelled as a sequence of people (a queue).
  • There may be any number of people in a queue, and any number of checkouts.
  • No one person may be in more than one queue at the same time.

Given Types

DeclarationWhat it isNotes
$[\text{Person}]$Basic typeAbstract set of all possible people

Key detail: each individual queue must be injective (no person twice in the same queue), and across all queues, no person may appear in two different queues.

The overall state is a sequence of queues. Each queue is an injective sequence of Person, and the whole structure is a sequence of such queues:

$$\text{seq}(\text{iseq}\ \text{Person})$$

Part (a) — Axiomatic Definition of State

Task: Using an axiomatic definition, define the set State, comprising all possible values of the sequence of sequences representing the checkouts. Each queue must be an injective sequence of Person, and no person may appear in two different queues.

Step 1 — Understand the task

We need to write an axiomatic definition. The declaration (State and its type) goes above the line. Below the line, we write predicates — invariants that every valid supermarket configuration must satisfy.

Step 2 — Identify the approach

We have two English constraints to encode:

  1. "No person appears twice in the same queue" → make each queue an injective sequence
  2. "No person appears in two different queues" → the ranges of any two distinct queues must be disjoint

Constraint 1 is handled at the type level by using iseq. Constraint 2 requires a separate predicate.

Step 3 — Plan the components

Type: The state is a member of P(seq(iseq Person)) — the power set ensures State is a set of valid configurations.

Disjointness predicate: ∀ s : State, ∀ i, j : dom s | i ≠ j, check that ran(s i) ∩ ran(s j) = ∅.

Part (b) — Function to Add a Person to a Queue

Task: Define a partial function $\text{add}$ that takes a queue number, a person, and a supermarket state, and returns a new state. The function models a person joining the back of a specified queue. Specify its signature, preconditions, and effect.

Step 1 — Understand the task

We need a function that modifies the supermarket state by adding one person to the back of one queue. The function takes three inputs: a queue number, a person, and the current state. It returns a new state. Since the function has preconditions (not every combination of inputs makes sense), it is a partial function.

Step 2 — Identify the approach

Three sub-problems to solve:

  1. Type signature: declare the input types and output type, using &pfun; because there are preconditions.
  2. Preconditions: two conditions must hold for the function to be defined — the queue must exist, and the person must not already be in any queue.
  3. Effect: use functional override ⊕ to replace one queue with the extended version, leaving the rest unchanged.

Step 3 — Plan the components

Signature: add : N × Person × State &pfun; State.

Precondition 1: n ∈ dom s — queue n must exist.

Precondition 2: p ∉ ran(⋃(ran s)) — person p must not be in any queue.

Effect: s ⊕ {n ↦ (s n)⌢⟨p⟩} — replace queue n with queue n plus p appended.

Build Step 3 — Precondition 2: person not in any queue

SymbolPlain EnglishWhat it does here
$p$bound variable — the person to addthe person we are checking
$\notin$not element of — a value does not belong to a setrequires $p$ to be absent from all queues
$\text{ran}$range — the set of all output valuesused twice: inner extracts queues, outer extracts people
$\bigcup$generalised union — merge all into onecombines all queues into one big relation

The second precondition says "person p is not already in any queue." We build this in layers:

  1. ran s = the set of all queues (each queue is a sequence, i.e. a function)
  2. ⋃(ran s) = union of all those functions into one big relation
  3. ran(⋃(ran s)) = the set of all people across all queues

Require p ∉ this set. Combined with precondition 1 using ∧:

$$\forall\, n : \mathbb{N};\ p : \text{Person};\ s : \text{State}\ |\ n \in \text{dom}\; s \wedge p \notin \text{ran}(\bigcup(\text{ran}\; s))\ \bullet\ \ldots$$

Why ran(⋃(ran s))?

  • $s$ is a sequence of queues, so $\text{ran}\; s$ gives the set of all queues (as sequences).
  • Each queue is a function $\{1,\ldots,k\} \to \text{Person}$. Taking $\bigcup$ merges all these functions into one big relation.
  • Taking $\text{ran}$ of that combined relation gives the set of all people appearing in any queue.
  • If $p$ is not in this set, $p$ is not in any queue — safe to add.

After the •, define the result. We want to replace queue n with the old queue plus p appended, leaving all other queues unchanged. Break it down piece by piece:

  1. s n = the current queue at position n (e.g. $\langle A, B \rangle$)
  2. (s n)⌢⟨p⟩ = append p to the end of that queue (e.g. $\langle A, B, p \rangle$)
  3. {n ↦ (s n)⌢⟨p⟩} = create a new mapping from index n to the extended queue
  4. s ⊕ {n ↦ ...} = override: replace the mapping at n, keep everything else

Putting it all together:

$$\text{add}(n, p, s) = s \oplus \{n \mapsto (s\,n) \frown \langle p \rangle\}$$

Complete answer

$$\text{add} : \mathbb{N} \times \text{Person} \times \text{State} \pfun \text{State}$$
$$\forall\, n : \mathbb{N};\ p : \text{Person};\ s : \text{State}\ |\ n \in \text{dom}\; s \wedge p \notin \text{ran}(\bigcup(\text{ran}\; s)) \bullet$$ $$\text{add}(n,\, p,\, s) = s \oplus \{n \mapsto (s\,n) \frown \langle p \rangle\}$$

"For every valid queue index n, person p not in any queue, and state s: the result of add is the state with queue n replaced by queue n with p appended at the back."
